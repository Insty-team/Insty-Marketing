## 3. 퇴사 기준: 언제가 합리적인가?

### 전문가/데이터 기반 기준

| 기준 | 구체적 수치 | 출처 |
|------|-----------|------|
| MRR ≥ 현 월급의 67~75% | MRR ≥ **300~334만원** | 다수 재무 어드바이저 |
| 3~6개월 연속 안정적 MRR | 일시적 스파이크 제외 | Founders Live |
| 비상 자금 3~6개월분 | **600~1,200만원** 현금 | Hustle to Startup |
| 사이드 시간이 성장 병목 | "시간만 더 있으면 2x 가능" | 경험적 |

### Sam 맞춤 퇴사 체크리스트

```
□ MRR ≥ 300만원 (3개월 연속)
□ MRR 성장률 양수 (전월 대비)
□ 비상 자금 1,000만원+ (현금)
□ 고객 이탈률(churn) < 5%/월
□ 사이드 시간이 명확한 성장 병목
□ 정규직 복귀 가능한 시장 상황 (SLAM 엔지니어 수요)
```

**6개 중 5개 이상 충족 시 퇴사 합리적.**

## 4. 시간 배분 전략

### Phase 1: 검증 (0~6개월) — 정규직 100% 유지

**주간 시간표:**

| 시간대 | 월~금 | 토 | 일 |
|--------|------|---|---|
| 09~18 | 정규직 | **사이드** | **사이드** |
| 19~21 | **사이드** | **사이드** | 쉬기 |
| 21~ | 학습/쉬기 | 쉬기 | 쉬기 |

- 평일 저녁: 2시간 × 5일 = 10시간
- 토요일: 8시간
- 일요일: 4시간
- **주당 22시간, 월 88시간**

**이 시간으로 할 수 있는 것:**
- MVP 개발: 4~8주
- 초기 고객 10명 확보: 4~8주
- PMF 검증: 8~16주

**이 시간으로 할 수 없는 것:**
- 복잡한 엔터프라이즈 제품 개발
- 적극적 영업/미팅 (정규직 시간과 충돌)

**Claude Code + AI 활용이 핵심.** AI가 개발 시간을 3~5배 단축해주므로, 88시간 × 3배 효율 = 264시간 상당.

### Phase 2: 성장 (6~18개월) — 정규직 유지, 강도 증가

- MRR 100만원 이상 확보
- 주당 사이드 시간을 25~30시간으로 증가 (정규직 효율화)
- 고객 피드백 기반 제품 개선
- 자동화로 운영 시간 최소화

### Phase 3: 전환 (18~24개월) — 퇴사 준비

- 퇴사 체크리스트 검토
- 3개월 런웨이 확보
- 정규직 퇴사 → 풀타임 전환
- 주당 50~60시간을 사이드에 투입

## 5. 리스크 관리

### 리스크 1: 번아웃

**현실:** 주 45시간 정규직 + 22시간 사이드 = 주 67시간 근무

**대응:**
- 일요일 오후는 반드시 쉬기
- 2주에 1번 완전 오프 (사이드도 안 함)
- AI 자동화로 반복 작업 최소화
- "빠르게 포기" — PMF 안 맞으면 3개월 내 pivot

### 리스크 2: 정규직 성과 하락

**대응:**
- 정규직에서 배우는 SLAM 지식을 사이드에 활용 (시너지)
- 정규직 업무 시간 내에서 효율 극대화
- 정규직에서의 인맥은 향후 B2B 영업에 활용

### 리스크 3: 사이드가 실패

**대응:**
- 정규직이 안전망 역할
- 3~6개월 단위로 검증, 안 되면 다른 아이디어로 전환
- Pieter Levels 모델: 한 제품 실패 → 다음 제품 (Photo AI는 Levels의 70번째+ 프로젝트)

### 리스크 4: 법적 이슈 (겸업 금지)

**확인 사항:**
- 근로계약서 겸업 조항 확인
- 한국 근로기준법상 근로시간 외 개인 활동은 원칙적 자유
- 경쟁사 제품이 아닌 이상 문제 가능성 낮음
- 사업자등록은 퇴사 이후 또는 사이드 매출이 커진 후

| 항목 | 상태 |

| 비상 자금 | ~3,000~4,500만원 |

**판단: 합리적.** MRR이 생활비 커버하고, 런웨이도 충분.

### 시나리오 C: 보수적 퇴사 (24개월+)

| 항목 | 상태 |

## 7. 최적 하이브리드 타임라인

```
Month 0~3: 검증 단계
├── 정규직 100% 유지
├── 사이드: 첫 번째 SaaS MVP 개발 (Insty Sales Pipeline SaaS)
├── 목표: 유료 고객 5명, MRR 50만원
└── AI 도구 최대 활용 (Claude Code, Cursor)

Month 4~6: PMF 탐색
├── 정규직 유지
├── 사이드: 고객 피드백 반영, 기능 개선
├── 목표: 유료 고객 20명, MRR 200만원
└── 안 되면: pivot (SLAM DevTools 또는 AgentFlow)

Month 7~12: 성장 단계
├── 정규직 유지 (단, 효율화)
├── 사이드: 제품 고도화, 두 번째 제품 시작
├── 목표: MRR 300~500만원
└── 퇴사 체크리스트 모니터링

Month 13~18: 전환 판단
├── 퇴사 체크리스트 5/6 이상 충족 → 퇴사
├── 미충족 → 정규직 유지, 사이드 지속
├── 비상 자금: 3,000만원+
└── 목표: MRR 500만원+

Month 19~24: 풀타임 전환 (퇴사 시)
├── 풀타임 SaaS 빌딩
├── 세 번째 제품 시작 (멀티 프로덕트)
├── 목표: MRR 2,000만원+
└── 투자/멘토 네트워크 구축
```

## 8. 결론

### "언제 정규직을 그만두는 것이 합리적인가?"

**답: MRR이 월급의 67% (300만원)를 3개월 연속 넘기고, 비상 자금 3,000만원이 있을 때.**

그전까지는 정규직이 최고의 투자다:
- 월 245만원 저축 → 비상 자금/투자 자본 축적
- SLAM 스킬 유지/향상
- 번아웃 없이 사이드 프로젝트 검증
- 실패해도 돌아갈 곳이 있음

### 정규직은 "족쇄"가 아니라 "런웨이"

정규직 월급 = 매달 자동으로 충전되는 런웨이.
이 런웨이 위에서 SaaS를 빌드하는 것이 가장 안전한 100억 전략.
